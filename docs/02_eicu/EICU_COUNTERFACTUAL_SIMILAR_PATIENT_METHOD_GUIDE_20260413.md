# 背景

当前 eICU 方向的核心目标已经明确收敛为一条主线：

- 给定一个新的 eICU Sepsis-3 患者窗口样本。
- 在历史病例中找到“医学状态最接近、且干预上可比较”的相似患者 donor。
- 基于 donor 原始方案、KG 修补方案和搜索层方案生成候选干预。
- 用模型预测这些候选方案作用在当前患者上的短期结果。
- 以结构化 JSON 和医生可直接查阅的 Markdown 输出结果。

经过九条修改后，系统已经不再是早期那种“只会找相似病例并抄方案”的原型，而是形成了一个完整的、可验证的“患者表征 + donor 检索 + 候选方案生成 + 反事实预测 + 风险门控 + 医生报告渲染”链路。

本文件用于替代旧版分散说明，统一回答以下问题：

- 当前方法到底是怎么工作的。
- 九条修改分别解决了什么问题。
- 当前方法的创新点、价值和局限性是什么。
- 单病例输入输出长什么样。
- 为什么第九条之后，系统会更保守、更可信，而不是看起来更“激进”。

# 本轮修改目标

本轮文档更新与结果整理的目标有三项：

1. 用一份总览文档完整介绍当前九条修改后的方法。
2. 明确输出目录中哪些结果属于“最新结果”与“可用于论文实验”的保留内容。
3. 用真实病例示例解释九条修改分别在哪些维度提升了结果，以及当前仍然存在的问题。

## 2. 当前整体方法

### 2.1 输入层

当前单病例推理支持两种输入方式：

1. 原始窗口输入
   - 由 `label_row`、`context_rows`、`future_target`、`metadata` 组成。
   - 这是当前最推荐的业务侧输入。
2. `forecast_sample` 直传模式
   - 直接传标准化后的 `sequence`、`patient_static`、`intervention_static`、`intervention_sequence`、`kg_features`。
   - 更适合内部调试，不建议作为默认外部接口。

输入经由 `build_eicu_sepsis3_inference_sample(...)` 统一构造为 `ForecastSample`，保证训练与推理使用同一套语义。

### 2.2 患者表征与事实预测

模型首先做两件事：

1. 对当前患者窗口做事实路径预测。
2. 在记忆空间中形成用于检索的患者表征。

这里的核心不是只看某一个时间点的 SOFA，而是同时综合：

- 当前窗口的序列趋势。
- 患者静态信息。
- 当前已经实施的干预。
- KG 映射得到的状态与指南信号。
- 动态病程摘要特征。

这一层决定后续 donor 检索是否“像对了人”，而不是只在数值上找一个近邻。

### 2.3 donor 检索与重排

系统不会只根据 embedding 相似度选 donor，而是用一套多维度 donor 评分：

- `donor_similarity`
- `donor_kg_similarity`
- `donor_guideline_compatibility`
- `donor_state_match`
- `donor_missing_care_penalty`
- `donor_contraindication_penalty`
- overlap 相关分数
- learned reranker 调整项

这一步的本质是：

- 不是“找最像的病人”。
- 而是“找既像、又可比、又在干预上不明显违背当前状态的病人”。

### 2.4 候选方案生成

当前候选方案不只来自 donor 原始方案，而是来自两层空间：

1. 安全层
   - `raw_intervention_store`
   - `generated_kg_repaired`
2. 搜索层
   - 把安全修补后的方案进一步扩展为一组小范围、可解释的模板变体。
   - 如更早抗菌、低强度升压支持、标准强度升压支持、bundle 组合等。

这一步让系统从“只能补洞”变成“可以在安全边界内比较多个候选治疗方向”。

### 2.5 反事实预测与候选选择

每个候选方案都会在当前患者上做一次反事实预测，得到：

- `predicted_delta`
- `candidate_selection_score`
- `predicted_counterfactual`

这里要特别强调：

- 当前方法仍然是候选干预模拟与风险重排框架。
- 它输出的是短期 proxy 预测差异。
- 它不是严格的因果疗效证明。

### 2.6 不确定性与辅助风险预测

系统已经把多任务预测和不确定性接到了单病例输出中，当前可给出：

- 未来 SOFA 主轨迹预测。
- 未来 SOFA 增量预测。
- 未来乳酸增量预测。
- 后续血管活性药需求概率。
- 后续呼吸支持升级概率。
- MC-dropout 形成的预测区间和改善上下界。

因此，当前输出不再只是“给一个分数”，而是同时给：

- 预测值。
- 不确定性。
- 多个临床辅助风险信号。

### 2.7 轻量 rolling-horizon

系统已支持两步轻量 rolling-horizon。

含义不是完整病程世界模型，而是：

- 第一步先为当前患者选方案。
- 把第一步方案的结果向前投影一小步。
- 在投影状态下重新检索 donor、重新比较候选方案。
- 统计两步累计改善和方案来源稳定性。

这使系统从“单步比较”推进到“短期连续比较”。

### 2.8 第九条后的严格门控

当前第九条引入的关键收口是：

- 单步 `predicted_delta <= 0` 时，不再允许直接输出推荐。
- 两步滚动折扣累计改善 `<= 0` 时，不再允许直接输出推荐。
- 改善下界低于阈值时，不再允许直接输出推荐。

这一步的本质是：

- 即使相似度很高，只要结果没有显示正向改善，就只能输出“仅供医生复核”。
- 系统不再把“像”错误等同于“值得直接采用”。

### 2.9 医生版最终输出

当前 Markdown 报告是由代码直接渲染生成，不是人工整理，也不是另一个外部大模型自由生成。

当前最终输出顺序已经调整为医生阅读顺序：

1. 病情判断
2. 干预建议
3. 短期趋势判断
4. 为什么暂不建议直接采纳
5. 相似患者证据
6. Top-3 备选方案对照
7. 医生复核重点
8. 方法边界说明
9. 选择理由

这意味着当前 `.md` 文件本身就是可交付结果，而不是“需要助手再解释一次”的中间产物。

## 3. 九条修改分别带来的提升

| 阶段 | 解决的问题 | 主要提升维度 | 为什么这算提升 |
| --- | --- | --- | --- |
| Phase 0: 分层评估基线 | 之前改动后不知道到底有没有变好 | 可验证性 | 先建立分层 baseline，后续每一步改动才有参照物 |
| Phase 2: factual / retrieval 双头拆分 | 找相似患者与做事实预测混在一起 | 检索语义纯度 | 患者状态相似与干预检索目标被拆开后，donor 更像“病情近邻”而不是“输出近邻” |
| Phase 3: 动态特征增强 | 静态摘要不足以描述病程变化 | 病程理解能力 | 模型会看趋势、波动、变化速度，而不只看某个时刻的数值 |
| Phase 4: 多任务预测 + 不确定性 | 只能给一个主目标，没有置信边界 | 风险表达能力 | 医生能同时看到主目标和辅助风险，还能看到区间而不是点预测 |
| Phase 5: donor 池扩大 + overlap 过滤 | donor 太少或虽然像但不可比 | donor 可比性 | 扩大 donor 池后召回更充分，overlap 过滤又避免“像但轨迹不连贯”的 donor 混进来 |
| Phase 6: 学习式 reranker | 规则分数不够细 | donor 排序质量 | 在规则基础上加入学习式重排，能更好地区分多个看起来都像的 donor |
| Phase 7: 安全层 + 搜索层 | 只会修补 donor 原方案，不会主动提出其他可行候选 | 候选空间丰富度 | 模型可以在安全边界内比较多种可解释方案，而不是单一路径 |
| Phase 8: rolling-horizon | 只能做当前一步的静态比较 | 连续决策能力 | 系统开始能回答“第一步这么做之后，第二步大概率还会不会沿同一路线走” |
| Phase 9: 严格门控 + 医生证据增强 | 结果可能过于乐观，报告也不够临床化 | 临床可信度与审计性 | 现在系统会把“不该直接推荐”的理由说清楚，并把证据强弱直接展示出来 |

## 4. 当前方法的创新点

当前方法最有价值的创新点，不是单一某个网络结构，而是以下组合：

### 4.1 把“相似患者检索”和“候选干预模拟”真正接起来

很多系统只能做到：

- 找相似病例。
- 或者直接预测风险。

当前方法把两件事连成了闭环：

- 先找到 donor。
- 再构造 donor 原方案、KG 修补方案和搜索方案。
- 再把这些方案真正作用到当前患者上做反事实预测。

### 4.2 把 KG 从“装饰性特征”变成 donor 评分和方案修补的一部分

KG 在当前系统里不只是一个附加输入，而是直接影响：

- donor 评分。
- 指南一致性。
- 状态匹配。
- 缺失照护惩罚。
- 候选方案修补。

### 4.3 把不确定性、辅助风险和短期滚动模拟一起纳入医生输出

这让系统不再只是一个“输出方案编号”的模型，而是一个能够解释：

- 为什么这么建议。
- 这个建议短期会怎样。
- 有多不确定。
- 为什么现在还不能直接采纳。

### 4.4 医生版输出已经代码化、模板化、可审计

这点很重要。

当前医生版 `.md` 文本不是助手再加工，而是系统结果的标准渲染产物。这样做的好处是：

- 可复现。
- 可审计。
- 不会因为额外 LLM 总结而产生漂移。

# 结果与分析

## 1. 当前系统已经具备的能力

九条修改后，当前系统已经能做到：

- 为新患者检索高相似 donor。
- 给出 donor 证据而不仅仅是一个最终推荐。
- 同时比较原始 donor、KG 修补方案和搜索层方案。
- 输出主目标预测、辅助风险预测、不确定性区间和两步滚动结果。
- 在结果不支持时自动降级为“仅供医生复核”。
- 直接生成医生视角 Markdown。

## 2. 用真实结果说明当前仍存在的问题

当前最适合说明“系统已经提升很多，但仍有局限”的例子是：

- `stay_141959`
- 输出文件：`output/analysis/phase9_physician_guardrail_141959.json`
- 医生版输出：`output/analysis/phase9_physician_guardrail_141959.md`

这个病例有一个非常典型的特征：

- 主 donor 相似度很高，约 `0.964`
- 指南一致性也不低，约 `0.710`
- 但模型单步 `predicted_delta = -0.058`
- 两步滚动折扣累计改善约 `-0.058`
- 改善下界约 `-0.161`

这说明当前系统已经能识别出一种很真实的临床机器学习难题：

- “这个病例和某个 donor 很像”
- 不等于
- “这个 donor 或其候选变体一定适合当前患者”

系统当前仍存在的问题正好体现在这个例子里：

1. 高相似 donor 不一定带来正向反事实收益。
2. 搜索层虽然能提出多个候选，但 top-3 全部仍为负收益。
3. 近邻 donor 整体平均指南一致性只有 `0.398`，说明“局部最优 donor 很强”不等于“邻域整体证据很强”。
4. 近邻 donor 的硬过滤通过率和 overlap 通过率都为 `0.0`，说明当前 donor 邻域仍存在“像，但不够可比”的问题。
5. 系统目前仍然是短期 proxy 模拟，而不是严格因果估计，因此不能把负向结果解释成真实疗效结论。

## 3. 用这一个结果解释九条修改分别带来的提升

仍然以 `stay_141959` 为例，可以看到九条修改不是都在“把结果变成正向”，而是在不同维度把结果变得更可靠。

### 第 1 条：分层评估基线

提升维度：

- 让我们知道当前看到的负结果是不是异常，而不是脚本坏了。

为什么是提升：

- 如果没有基线，`stay_141959` 这种“高相似但负收益”的结果会让人分不清是模型问题，还是结果本来就应该谨慎。

### 第 2 条：factual / retrieval 双头拆分

提升维度：

- donor 更像“病情近邻”。

为什么是提升：

- 当前主 donor 相似度能达到 `0.964`，说明 donor 检索已经明显更稳定地聚焦在状态相似患者，而不是混入太多“只是在输出上接近”的病例。

### 第 3 条：动态特征增强

提升维度：

- 更好理解病程变化。

为什么是提升：

- 在这个病例里，系统不仅知道它是脓毒症相关状态，还能识别其模式为“尖峰/突变 + 阶段转换”，并把“血流动力学支持偏弱”识别为当前关键缺口。这比只看单个 SOFA 数字更接近临床判断。

### 第 4 条：多任务预测 + 不确定性

提升维度：

- 不再只给一个结论，而是给一组短期风险信号和区间。

为什么是提升：

- 当前结果中，不仅知道主轨迹不改善，还能看到：
  - 未来 SOFA 增量略低
  - 但后续血管活性药需求概率和呼吸支持升级概率略高
  - 改善下界仍为负
- 这说明系统已经不是“单指标乐观化”，而是能把相互冲突的短期趋势暴露出来。

### 第 5 条：donor 池扩大 + overlap 过滤

提升维度：

- 不再只靠一个 donor，开始看 donor 邻域。

为什么是提升：

- 当前输出会明确告诉你：
  - 近邻 donor 数量是 3
  - 平均相似度 `0.873`
  - 但 overlap 通过率 `0.0`
- 这意味着系统已经能告诉医生“虽然找到了像的 donor，但邻域整体可比性仍然一般”，而不是只报一个最优 donor 就结束。

### 第 6 条：学习式 reranker

提升维度：

- 多个候选 donor 之间的优先级更细。

为什么是提升：

- 在高相似病例中，经常会出现多个 donor 都“看起来能用”。学习式 reranker 的价值不是保证一定变正，而是让排序更贴近综合证据，减少简单规则打分带来的粗糙误排。

### 第 7 条：安全层 + 搜索层

提升维度：

- 候选方案空间更丰富。

为什么是提升：

- 当前 top-3 候选方案已经不止一个：
  - 低强度升压支持模板
  - 脓毒症 bundle
  - 更早抗菌模板
- 虽然它们在这个病例里都没有带来正向收益，但系统至少已经具备“提出多个可解释备选方案并逐一比较”的能力，这比过去只能输出一个修补版 donor 方案强很多。

### 第 8 条：rolling-horizon

提升维度：

- 可以做短期连续比较，而不是只看第一步。

为什么是提升：

- 当前结果显示：
  - 两步滚动折扣累计改善仍为负
  - 方案来源保持稳定
- 这让我们知道问题不是“第一步偶然失真”，而是短期连续路径也没有显示正向。这个信息对临床复核非常关键。

### 第 9 条：严格门控 + 医生证据增强

提升维度：

- 临床可信度与报告可审计性。

为什么是提升：

- 如果没有第九条，这个病例很可能会被写成“相似度高，因此值得考虑”。
- 现在系统会明确输出：
  - 状态：`仅供医生复核`
  - 原因：单步不改善、两步不改善、下界仍为负
- 这说明第九条带来的提升不是“让结果更好看”，而是“让结果更真实、更负责任”。

## 4. 当前经过九条修改后仍然存在的问题

当前系统已经明显成熟，但仍有以下问题：

1. 仍然缺乏严格 OPE/DR 级别的因果评估。
2. rolling-horizon 仍是轻量近似，不是完整病程世界模型。
3. donor 邻域整体证据有时仍弱于最佳 donor 个体证据。
4. 搜索层虽然扩展了候选空间，但候选空间仍是模板化的，尚未覆盖更复杂的组合干预。
5. 当前主目标仍以短期 SOFA proxy 为核心，不能替代更长期、更临床结局化的评估。
6. 医生版输出已经可查阅，但尚未形成真正的 bedside workflow，例如签核、复核意见回写、与真实医嘱系统对接。

# 已知问题

- 当前方法仍属于“候选干预模拟 + 风险重排”，不是严格因果疗效证明。
- 高 donor 相似度不代表高治疗可迁移性。
- donor 邻域整体一致性仍有提升空间。
- 搜索层候选虽然更丰富，但仍偏模板化。
- rolling-horizon 只是轻量两步近似，不能替代长期策略学习。
- 当前对“为什么不推荐”的解释已经可读，但对“如果一定要继续优化，下一步应该怎么试”仍偏工程视角，尚未完全临床工作流化。

# 下一步计划

1. 引入更严格的 OPE / DR 设计，但要明确适用前提和边界。
2. 把 donor 邻域一致性真正纳入候选排序，而不是只放在报告层展示。
3. 扩展搜索层，让候选方案不只来自固定模板。
4. 把 rolling-horizon 从两步轻量近似推进到更稳定的短期策略评估。
5. 增加“医生复核后反馈”回写接口，把医生判断反哺到后续记忆和排序中。
6. 在当前医生版 Markdown 基础上进一步沉淀成论文图表、病例附录模板和答辩材料入口。
