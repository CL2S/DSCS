# 脓毒症预测知识经验库设计方案

**——基于外部层级记忆思想的动态临床知识系统**

## 1. 文档目的

本文档旨在将论文中“外部层级记忆增强模型”的核心思想，转化为一套面向**脓毒症预测**场景的“知识经验库”设计方案。该方案不把知识库仅仅视为静态文献或规则仓库，而是将其设计为一个能够：

* 存储静态医学知识；
* 积累历史病例经验；
* 动态维护当前患者时序状态摘要；
* 支持模型读写交互；
* 为风险预测提供可解释证据链；

的**动态记忆系统**。

该系统适用于 ICU、急诊、普通病房等连续监测场景，可服务于脓毒症早筛、风险分层、恶化预警和辅助解释。

---

# 2. 设计背景与核心思想

## 2.1 问题本质

脓毒症预测不是一个只依赖当前时刻数据的分类任务，而是一个典型的**部分可观测、长时依赖**决策问题。

对于同一个当前状态，例如：

* MAP 下降；
* 乳酸升高；
* 呼吸频率增加；
* 白细胞异常；

其临床意义往往取决于更长时间跨度内的历史轨迹，例如：

* 低血压是否持续存在；
* 抗生素后是否改善；
* 液体复苏后灌注是否恢复；
* 器官功能是否呈进行性恶化；
* 早先是否已有感染灶证据。

因此，单纯依赖某个固定窗口、或者把全部历史无差别堆入模型，都会带来问题：

* 固定短窗：会丢失关键远程信息；
* 无限堆历史：计算成本高，噪声多，且不利于解释。

## 2.2 从论文中提炼出的可迁移思想

可借鉴的不是原论文的强化学习任务形式，而是其**记忆设计原则**：

### （1）记忆要分层，而不是所有信息混在一起

不同抽象层级的信息应分开管理，例如原始事件、状态摘要、决策解释。

### （2）系统必须同时支持“读历史”和“写历史”

模型预测时要读取长期记忆；新时窗到来时又要能把新证据写回系统。

### （3）记忆要有容量边界

不追求保存一切，而要保存最关键、最能支持后续预测的摘要。

### （4）记忆条目必须带时间锚点

医疗信息的价值高度依赖“发生了多久”，而不仅是“发生了什么”。

### （5）记忆更新不能是简单覆盖

旧信息应衰减但不应立刻消失，应保留一定历史影响。

---

# 3. 总体设计目标

本知识经验库的目标包括：

1. **支撑脓毒症风险预测**
   向预测模型提供多时间尺度、结构化、动态更新的患者状态记忆。

2. **融合规则知识与病例经验**
   同时接入指南、共识、临床规则以及相似历史病例模式。

3. **支持连续写回与滚动维护**
   每个时间窗结束后自动提炼关键信息并更新患者记忆。

4. **提升可解释性**
   对每次高风险预警给出证据链和时间轨迹解释。

5. **支持多模型接口**
   可对接时序深度学习模型、规则引擎、检索增强模型和多模态预测模型。

---

# 4. 总体架构设计

整个系统建议设计为“三层库 + 一套读写机制 + 一层模型接口”的结构。

## 4.1 三层库结构

### 第一层：静态医学知识库（Static Knowledge Base）

用于存储相对稳定、通用的临床知识，包括：

* Sepsis-3 定义；
* SOFA/qSOFA/SIRS 规则；
* 感染相关诊疗共识；
* 液体复苏原则；
* 血流动力学管理知识；
* 乳酸监测与清除相关知识；
* 抗菌药经验治疗策略；
* 器官功能恶化判据。

该层的特点是：
**更新频率低、权威性强、适合作为规则与解释依据。**

---

### 第二层：病例经验库（Case Experience Base）

用于存储历史患者的轨迹经验与关键转折模式，例如：

* 典型脓毒症进展路径；
* 高风险前驱模式；
* 干预后改善/无改善模式；
* 感染灶到器官功能恶化的常见链条；
* 相似患者的演化摘要；
* 成功/失败处置经验片段。

该层的特点是：
**面向相似病例检索，强调“这个病人像谁、历史上类似情况如何发展”。**

---

### 第三层：个体动态记忆库（Patient Dynamic Memory）

这是本方案最关键的一层。
它不保存全部原始监测数据，而是为每位患者维护一组**动态记忆单元**，用于表示当前患者在近时段和长时段上的重要状态摘要。

例如：

* 最近 6 小时是否持续低血压；
* 乳酸是否持续升高；
* 是否有新发肾功能损伤；
* 是否已开始抗菌药；
* 是否在补液后仍低灌注；
* 是否存在疑似感染证据；
* SOFA 是否持续上升。

该层的特点是：
**动态写入、持续更新、服务当前预测，是患者级长期工作记忆。**

---

# 5. 各库内容设计

## 5.1 静态医学知识库设计

### 5.1.1 建议知识单元类型

1. 诊断定义类
2. 风险评分规则类
3. 指南推荐类
4. 临床路径类
5. 治疗原则类
6. 证据解释类
7. 警报触发类

### 5.1.2 示例条目

* “疑似感染 + SOFA 增加 ≥ 2，支持脓毒症判定”
* “持续乳酸升高提示组织低灌注风险”
* “液体复苏后 MAP 仍低提示循环支持不足”
* “尿量下降可提示早期肾灌注异常或 AKI 风险”

| -------------------- | --------------- |

| content              | 条目正文            |
| trigger_conditions   | 适用触发条件          |
| related_variables    | 关联变量，如乳酸、MAP、尿量 |
| evidence_level       | 证据等级            |

| version              | 版本号             |

| explanation_template | 面向解释输出的模板       |

## 5.2 病例经验库设计

### 5.2.1 经验单元设计原则

病例经验库不应只是“存整个病历”，而应提炼为可检索、可比较的经验片段。
建议按以下粒度组织：

* 患者级摘要；
* 阶段级摘要；
* 关键转折事件片段；
* 干预-响应片段；
* 风险升级片段。

### 5.2.2 示例经验片段

* “感染证据出现后 8 小时内乳酸持续上升，12 小时后出现休克”
* “补液后 MAP 未改善，后续需升压药支持”
* “抗生素启动后 6 小时乳酸下降，风险缓解”
* “呼吸支持升级后伴肾功能恶化，提示多器官受累”

### 5.2.3 推荐字段

| 字段名                  | 含义                 |
| -------------------- | ------------------ |
| case_unit_id         | 经验单元编号             |
| patient_group        | 病例分组标签             |
| stage                | 所处阶段，如感染早期、恶化期、休克期 |
| summary_text         | 经验摘要               |
| feature_signature    | 特征签名向量或结构化标签       |
| time_pattern         | 时间模式描述             |
| intervention_pattern | 干预模式               |
| response_pattern     | 干预响应模式             |
| outcome_label        | 结局标签               |
| similarity_tags      | 相似性标签              |
| source_case_id       | 来源病例编号             |
| confidence_score     | 经验可信度              |
| created_at           | 创建时间               |

## 5.3 个体动态记忆库设计

## 5.3.1 设计原则

个体动态记忆库不是简单缓存，而是类似“患者长期状态摘要板”。

每个患者维护若干个**记忆槽（memory slots）**，每个槽代表一个重要临床维度。建议按临床语义分槽，而不是随机分槽。

### 建议槽位

1. 感染证据槽
2. 循环灌注槽
3. 呼吸状态槽
4. 肾功能与尿量槽
5. 炎症与代谢槽
6. 器官功能趋势槽
7. 治疗干预槽
8. 干预响应槽
9. 风险解释槽
10. 未分类关键事件槽

---

## 5.3.2 每个记忆槽的字段设计

| 字段名              | 含义           |
| ---------------- | ------------ |
| memory_id        | 记忆条目编号       |
| patient_id       | 患者编号         |
| slot_type        | 槽位类型         |
| summary          | 当前摘要内容       |
| state_vector     | 结构化状态表示      |
| anchor_time      | 最近一次关键更新时间   |
| start_time       | 该状态首次出现时间    |
| end_time         | 最近一次有效时间     |
| source_events    | 支撑该记忆的原始事件列表 |
| importance_score | 当前重要度        |
| confidence_score | 当前置信度        |
| freshness_score  | 新鲜度          |
| decay_factor     | 衰减参数         |
| status           | 激活/弱化/待淘汰    |
| evidence_chain   | 简要证据链        |
| last_access_time | 最近被读取时间      |
| update_count     | 更新次数         |
| version          | 版本号          |

---

### 示例 1：循环灌注槽

* slot_type：hemodynamic

# 6. 字段标准化设计建议

为保证后续检索、更新和模型接入，建议字段同时包含三类表示：

## 6.1 文本摘要字段

用于可解释输出和自然语言检索，例如：

* “近 6 小时尿量下降并伴肌酐升高”
* “抗生素后循环未改善”

## 6.2 结构化字段

用于规则判断和统计建模，例如：

* MAP_min
* Lactate_trend
* Urine_output_6h
* SOFA_delta
* Antibiotic_started
* Fluid_response_flag

## 6.3 向量字段

用于语义检索和相似病例匹配，例如：

* summary_embedding
* state_embedding
* event_embedding_sequence

# 7. 更新规则设计

这是整个系统最重要的部分之一。

## 7.1 为什么更新规则比存储更重要

如果没有合理更新机制，经验库会迅速退化为：

* 信息冗余；
* 陈旧条目堆积；
* 噪声覆盖有效状态；
* 模型读到不相关历史。

因此必须设计一套“**有边界的动态更新规则**”。

---

## 7.2 更新触发时机

建议以下情况触发更新：

### （1）固定时间窗触发

例如每 1 小时、每 2 小时更新一次。

### （2）关键事件触发

当出现以下事件时立即触发：

* 乳酸显著升高；
* MAP 持续低；
* 尿量骤降；
* 新启升压药；
* 新启抗菌药；
* SOFA 快速上升；
* 疑似感染证据新增。

### （3）预测前触发

模型出风险预测前先做一次“最新状态写回”，确保记忆库反映当前情况。

---

## 7.3 写入流程

每个新时间窗到来后，执行以下步骤：

### 步骤 1：提取窗口内关键事件

从原始时序数据中提取：

* 异常生命体征；
* 检验指标变化；
* 干预事件；
* 风险评分变化；
* 器官功能变化。

### 步骤 2：生成窗口级状态摘要

将本窗口的信息压缩成：

* 文本摘要；
* 结构化状态；
* 向量表达。

### 步骤 3：匹配对应槽位

判断该摘要应更新到哪个记忆槽：

* 感染类 → infection slot
* 循环类 → hemodynamic slot
* 呼吸类 → respiratory slot
* 干预响应类 → response slot

### 步骤 4：决定是新增、融合还是替换

根据相似度、时效性和重要性决定更新方式。

---

## 7.4 更新策略

## 7.4.1 新增策略

满足以下条件时新增记忆条目：

* 当前槽为空；
* 出现了全新临床状态；
* 原记忆无法覆盖该新现象；
* 新事件具有较高临床重要度。

例如：

* 新发乳酸升高；
* 新启升压药；
* 新出现感染灶证据。

---

## 7.4.2 融合更新策略

当新窗口信息与现有记忆属于同一临床状态，但内容有新变化时，采用融合更新。

### 融合公式思想

可使用类似以下更新方式：

**new_memory = λ × new_info + (1 - λ) × old_memory**

其中：

* λ 大：更重视新证据；
* λ 小：更保留旧状态。

### 医疗场景中 λ 的设计建议

不同类型信息应使用不同 λ：

* 血压、SpO2、呼吸频率：λ 较大，强调近期变化；
* 乳酸趋势：λ 中等；
* 感染证据：λ 中等偏小；
* 耐药史、既往严重感染史：λ 更小。

---

## 7.4.3 替换策略

以下情况可考虑替换旧条目：

* 条目长期未再被支持；
* 条目重要度已很低；
* 条目与当前临床状态显著不一致；
* 槽位容量有限且出现更高价值新信息。

---

## 7.4.4 淘汰策略

建议引入类似 LRU + 医疗重要度的淘汰逻辑，而不是纯时间淘汰。

### 记忆淘汰评分示例

可定义：

**evict_score = a × stale_score + b × low_importance + c × low_relevance + d × low_access**

分数越高越优先淘汰。

其中：

* stale_score：陈旧程度；
* low_importance：临床重要度低；
* low_relevance：与当前风险关系弱；
* low_access：长时间未被读取。

---

# 8. 遗忘与衰减机制设计

## 8.1 为什么需要遗忘机制

临床信息不是永久同等有效。
例如：

* 10 分钟前 MAP 低与 24 小时前 MAP 低意义不同；
* 2 小时前乳酸上升比 2 天前更能反映当前风险；
* 既往感染证据虽然老，但也不能完全无视。

因此记忆系统需要“**逐渐衰减，而非瞬间消失**”。

---

## 8.2 建议的衰减字段

每个记忆条目可维护：

* freshness_score：越新越高；
* decay_factor：类型特异衰减率；
* half_life_hours：半衰期；
* active_weight：当前在检索和模型中的有效权重。

---

## 8.3 不同信号的衰减建议

### 短半衰期信息

* MAP
* HR
* RR
* SpO2
* 当前升压药剂量变化

### 中半衰期信息

* 乳酸趋势
* 尿量趋势
* SOFA 变化
* 呼吸支持升级

### 长半衰期信息

* 感染灶证据
* 培养阳性信息
* 耐药风险
* 近期脓毒症史
* 免疫抑制背景

---

# 9. 检索逻辑设计

知识经验库不是被动存储，而要支持多源检索。

## 9.1 检索目标

在每次预测时，系统应能从三个层面检索：

1. 当前患者最相关的动态状态记忆；
2. 与当前状态最相似的历史病例经验；
3. 与当前状态匹配的静态医学知识。

---

## 9.2 检索流程

## 9.2.1 第一步：构建当前查询表示

从当前时窗数据生成查询对象，包括：

* 当前状态文本摘要；
* 当前结构化特征；
* 当前时间上下文；
* 当前任务目标（脓毒症风险预测/解释/分层）。

---

## 9.2.2 第二步：检索个体动态记忆

优先读取当前患者的动态记忆库。

### 检索依据

* 内容相似度；
* 槽位类型匹配；
* 时间相对距离；
* 当前风险任务相关性；
* 最近访问与重要度。

### 检索结果示例

* 近 4 小时持续低灌注；
* 6 小时前感染证据增强；
* 抗生素启动后无明显改善；
* SOFA 持续增加。

---

## 9.2.3 第三步：检索病例经验库

用当前状态去匹配相似病例轨迹或经验片段。

### 检索依据

* 状态签名相似度；
* 时间轨迹相似度；
* 干预模式相似度；
* 当前阶段相似度。

### 返回内容

* 相似病例摘要；
* 历史演化趋势；
* 常见结局；
* 关键转折事件。

---

## 9.2.4 第四步：检索静态知识库

根据当前证据链匹配规则知识和解释知识。

### 示例

如果当前患者表现为：

* 疑似感染；
* SOFA 增加；
* 乳酸上升；
* MAP 下降；

则可召回：

* Sepsis-3 定义条目；
* 低灌注相关解释条目；
* 液体复苏后评估原则条目；
* 器官功能恶化风险说明条目。

---

# 10. 检索排序机制

## 10.1 排序原则

建议综合以下因素排序：

* 语义相关性；
* 时间相关性；
* 任务相关性；
* 临床重要性；
* 置信度；
* 新鲜度。

---

## 10.2 排序分数示例

可定义：

**final_score = w1 × semantic_similarity + w2 × temporal_relevance + w3 × clinical_importance + w4 × confidence + w5 × task_match**

其中：

* semantic_similarity：内容匹配；
* temporal_relevance：时间匹配；
* clinical_importance：临床重要度；
* confidence：证据可信度；
* task_match：与当前预测目标一致性。

---

## 10.3 时间相关性设计

这里建议重点借鉴“相对时间偏置”的思想。

检索时不能只比较内容，还要比较“距离当前多远”：

* 抗生素开始后 2 小时仍低血压；
* 液体后 4 小时乳酸不降；
* 12 小时前已有感染迹象；
* 24 小时内 SOFA 上升。

也就是说，检索排序中要把**相对时间关系**作为独立特征，而不是只看绝对时间戳。

# 11. 模型接口设计

本知识经验库的价值，最终体现在它如何服务模型。

## 11.1 输入接口

预测模型在每次推理时接收两类输入：

### （1）原始当前时序输入

包括：

* 当前时间窗生命体征；
* 检验指标；
* 干预记录；
* 用药事件；
* 评分特征。

### （2）外部记忆输入

从知识经验库检索得到：

* 当前患者动态记忆条目；
* 相似病例经验片段；
* 静态医学知识条目。

---

## 11.2 输出接口

模型输出应至少包括：

1. 脓毒症风险分数；
2. 风险等级；
3. 触发风险的关键证据；
4. 需要写回知识库的新增摘要；
5. 可解释证据链。

---

## 11.3 读接口（Memory Read API）

### 功能

在预测前读取当前最相关记忆。

### 输入

* patient_id
* current_time
* query_summary
* task_type
* top_k

### 输出

* dynamic_memory_items
* similar_case_items
* static_knowledge_items

### 示例接口

```json
{
  "patient_id": "P001",
  "current_time": "2026-03-11T10:00:00",
  "task_type": "sepsis_risk_prediction",
  "top_k": 10
}
```

返回：

```json
{
  "dynamic_memory_items": [...],
  "similar_case_items": [...],
  "static_knowledge_items": [...]
}
```

---

## 11.4 写接口（Memory Write API）

### 功能

将当前窗口提炼出的新状态写回个体动态记忆库。

### 输入

* patient_id
* window_start
* window_end
* extracted_events
* summary_text
* structured_state
* slot_type
* importance_score

### 输出

* update_type（新增/融合/替换）
* updated_memory_id
* new_version

### 示例

```json
{
  "patient_id": "P001",
  "window_start": "2026-03-11T08:00:00",
  "window_end": "2026-03-11T10:00:00",
  "slot_type": "hemodynamic",
  "summary_text": "近2小时MAP持续低于65，补液后改善有限",
  "importance_score": 0.91
}
```

---

## 11.5 解释接口（Explanation API）

### 功能

输出预测依据。

### 返回内容

* 当前高风险的主要证据；
* 哪些证据来自当前窗口；
* 哪些来自长期记忆；
* 哪些规则知识支持当前解释；
* 哪些相似病例支持当前判断。

### 示例输出

```json
{
  "risk_score": 0.89,
  "evidence_chain": [
    "疑似感染证据增强",
    "SOFA较基线上升",
    "乳酸持续升高",
    "补液后低灌注未明显改善"
  ],
  "supporting_knowledge": [
    "Sepsis-3定义",
    "低灌注相关解释规则"
  ],
  "similar_case_patterns": [
    "病例A：感染后8小时乳酸升高并进展为休克"
  ]
}
```

# 12. 预测流程设计

建议整体运行流程如下：

## 第一步：采集当前时间窗数据

从 EHR、监护仪、检验系统、医嘱系统等获取最新数据。

## 第二步：事件抽取与状态摘要

提取异常事件、干预信息和趋势信息，形成当前窗口摘要。

## 第三步：读取知识经验库

检索当前患者动态记忆、相似病例经验和静态知识条目。

## 第四步：融合输入并进行预测

模型综合当前数据 + 外部记忆，输出风险分数与解释。

## 第五步：写回动态记忆库

将本窗口重要信息写回患者记忆槽。

## 第六步：生成解释与告警

向临床侧输出风险级别、关键证据与建议关注点。

# 13. 可解释性设计

脓毒症预警系统如果只输出一个概率，临床接受度会很低。
因此建议系统输出以下解释结构：

## 13.1 风险证据链

例如：

* 当前存在感染证据；
* SOFA 上升；
* 乳酸升高；
* 血流动力学未改善。

## 13.2 时间轨迹解释

例如：

* 6 小时前首次出现感染迹象；
* 4 小时前开始低血压；
* 2 小时前补液后仍未改善；
* 当前乳酸仍高。

## 13.3 长短期证据区分

例如：

* 当前证据：MAP 低、乳酸高；
* 长期证据：感染持续存在、SOFA 已连续上升。

## 13.4 对应知识支撑

例如：

* 符合 Sepsis-3 相关定义；
* 符合持续低灌注高风险模式；
* 相似病例中该模式常对应恶化。

# 16. 风险与注意事项

## 16.1 数据质量问题

医疗数据普遍存在：

* 缺失；
* 异步采样；
* 测量噪声；
* 记录延迟；
* 干预偏差。

因此动态记忆库不能机械更新，必须考虑置信度。

## 16.2 标签定义问题

脓毒症标签定义在不同数据集和机构间可能不同。
知识库设计要尽量与标签解耦，优先围绕“证据链”和“状态演化”组织。

## 16.3 避免过度堆积

系统不是“保存越多越好”，而是“保留越关键越好”。

## 16.4 避免黑箱更新

记忆更新规则一定要可审计，否则临床侧难以信任。

# 17. 最终方案总结

本方案的核心，不是做一个普通数据库，而是做一个**面向脓毒症连续预测的动态知识经验系统**。

它的本质特征是：

* **三层结构**：静态知识、病例经验、个体动态记忆；
* **分层记忆**：不同抽象层级的信息分别维护；
* **时间锚定**：每条经验都带时间意义；
* **读写闭环**：预测前读历史，预测后写回新摘要；
* **容量受控**：不保存全部历史，只保留关键状态；
* **渐进遗忘**：旧信息衰减但不瞬时消失；
* **面向解释**：每次高风险都能给出证据链。

这套设计最重要的价值，是把脓毒症预测从“单次打分”升级为“**基于动态临床记忆的连续风险理解**”。
